# TriviumDB 修复优先级清单

这份清单将当前的架构级 review 收敛为一份可执行的修复计划。

## P0：必须立即修复

### 1. 防止 `Mmap` 模式下 `.tdb` / `.vec` 跨文件撕裂

- 风险：
  `save_mmap()` 会先写入并替换 `.vec`，再更新 `.tdb`；如果中途崩溃，可能留下“新 `.vec` + 旧 `.tdb`”的组合。
- 影响：
  重启后出现静默的 payload / 向量错配。
- 相关代码：
  - `src/storage/file_format.rs`
  - `save_mmap()`
  - `load()`
- 修复方向：
  - 为 `.tdb` 和 `.vec` 引入共享的代际标记或 manifest。
  - 加载时校验两个文件是否属于同一次 flush。
  - 考虑对这两个文件实现两阶段提交，而不是把它们视为各自独立提交。
- 验收标准：
  - 在 `.vec` 替换与 `.tdb` 替换之间崩溃时，数据库不能以混合状态被成功读取。
  - 加载器必须拒绝代际不匹配的文件组合。

### 2. 让 WAL 回放具备幂等性，并正确推进 `next_id`

- 风险：
  `replay_entry()` 调用的是 `raw_insert()`，它会直接追加内部状态，不检查重复 ID，也不会推进 `next_id`。
- 影响：
  恢复后可能出现重复内部记录、ID 被复用、持久状态被污染。
- 相关代码：
  - `src/database.rs`
  - `src/storage/memtable.rs`
- 修复方向：
  - 增加一个“恢复安全”的插入路径，要求它：
    - 遇到已存在 ID 时跳过或做一致性协调，
    - 更新 `next_id = max(next_id, id + 1)`，
    - 保持内部索引结构一致。
  - 明确 WAL 回放到底要实现“严格回放”“幂等回放”还是“补偿式回放”，并据此落代码。
- 验收标准：
  - 同一份已提交 WAL 回放两次，结果必须与回放一次完全一致。
  - 从 `.tdb + WAL` 恢复后，后续插入不能复用已经物化过的节点 ID。

### 3. 恢复真正的事务持久化语义

- 风险：
  `Transaction::commit()` 会先修改内存态，再写 WAL。
- 影响：
  一旦 WAL 写入失败，当前进程里的状态和重启后的状态会分叉。
- 相关代码：
  - `src/database.rs`
  - `Transaction::commit()`
- 修复方向：
  - 二选一：
    - 先写 WAL，再应用到 memtable；
    - 或先在 shadow state 上执行，等 WAL 成功后再发布到正式状态。
  - 同时更新注释和文档，使其与真实语义一致。
- 验收标准：
  - 提交失败时，运行时状态和重启后的状态都必须保持未变。
  - 断电场景和 WAL 写入失败场景必须有测试覆盖。

## P1：在更广泛发布前修复

### 4. 确保被删除节点不能再被更新

- 风险：
  `delete()` 删除了 payload，但保留了 `ids_to_indices`；`update_vector()` 仍然会成功。
- 影响：
  tombstone 仍可被修改，删除语义被破坏。
- 相关代码：
  - `src/storage/memtable.rs`
- 修复方向：
  - 更新写路径，判断节点是否“存活”不能只看内部索引，还要看逻辑存在性。
  - 审计所有依赖 `ids_to_indices` 的读写逻辑。
- 验收标准：
  - 对已删除节点调用 `update_vector()` 和 `update_payload()` 时都必须稳定失败。

### 5. 修复 BQ 粗筛在 delete / update 后的正确性问题

- 风险：
  BQ 签名只在向量总数变化时重建，而 BQ 搜索又使用 `0..node_count()` 扫描，而不是扫描完整内部槽位。
- 影响：
  删除或更新之后，签名会过期，活节点也可能被漏扫。
- 相关代码：
  - `src/storage/memtable.rs`
  - `src/database.rs`
- 修复方向：
  - 在向量更新和逻辑删除时失效并重建 BQ 签名。
  - 遍历活跃的内部条目，而不是简单用 `0..node_count()`。
- 验收标准：
  - 在包含删除和更新的固定测试数据上，开启 BQ 的搜索结果应与 brute force 的候选结果一致。

### 6. 统一查询解析器与执行器的语义

- 风险：
  Parser 接受匿名节点模式，但 Executor 在路径扩展时要求当前节点必须有变量名。
- 影响：
  一些看起来合法的查询会静默返回空结果。
- 相关代码：
  - `src/query/parser.rs`
  - `src/query/executor.rs`
- 修复方向：
  - 二选一：
    - 在 Executor 中真正支持匿名中间节点遍历；
    - 或在 Parser 层明确拒绝当前不支持的匿名写法。
  - 优先用“解析时报错”，不要让它静默失败。
- 验收标准：
  - 匿名节点模式的行为必须被明确定义，并由测试覆盖。

### 7. 持久化文本索引，或在重启后自动重建

- 风险：
  `TextIndex` 目前完全驻留内存，重启后会被重新初始化。
- 影响：
  文本混合检索在重启后会静默退化。
- 相关代码：
  - `src/storage/memtable.rs`
  - `src/index/text.rs`
  - `src/storage/file_format.rs`
- 修复方向：
  - 要么持久化文本索引产物；
  - 要么持久化足够的原始状态，使其在 open 时自动重建。
  - 如果短期内不做，也要把它明确写成限制项，而不是隐含行为。
- 验收标准：
  - 数据库重启后，文本混合检索行为必须可预测且有文档说明。

## P2：重要的工程加固

### 8. 补齐恢复与持久化测试覆盖

- 当前缺失的覆盖范围：
  - WAL 回放
  - `Rom` / `Mmap` 两种模式下的 flush/load
  - 撕裂写入模拟
  - delete/update 边界行为
  - BQ 正确性
  - 查询 DSL 行为
- 建议新增测试文件：
  - `tests/recovery.rs`
  - `tests/file_format.rs`
  - `tests/transaction.rs`
  - `tests/query.rs`
- 验收标准：
  - 每一个 P0 / P1 项都至少对应一个回归测试。

### 9. 统一 Rust / Python / Node 的包元数据

- 风险：
  当前多个包清单里的版本号和 license 声明不一致。
- 影响：
  发布信息混乱，事故定位与版本追踪成本更高。
- 相关代码：
  - `Cargo.toml`
  - `pyproject.toml`
  - `package.json`
- 修复方向：
  - 统一版本号和 license 声明。
  - 视情况引入单一发布源。

### 10. 重新评估数据库级锁模型

- 风险：
  搜索过程中长时间持有 `Mutex<MemTable>`，会限制并发能力。
- 影响：
  在读写混合负载下容易形成吞吐瓶颈。
- 相关代码：
  - `src/database.rs`
  - `src/index/brute_force.rs`
- 修复方向：
  - 在正确性问题修完后，再考虑 `RwLock`、不可变快照或更细粒度锁范围。

## 推荐执行顺序

1. 先修复 `.tdb` / `.vec` 的持久化一致性。
2. 再修复 WAL 回放语义和 `next_id`。
3. 修复事务提交的持久化顺序。
4. 锁定 delete / update 的基本不变量。
5. 修复 BQ 正确性。
6. 修复查询 DSL 的语义契约不一致。
7. 决定文本索引的持久化策略。
8. 为以上每项补回归测试。
9. 清理发布元数据。
10. 在正确性稳定后再做并发优化。

## 发布门槛

在所有 P0 项，以及第 4、5、6、8 项没有完成之前，不建议把它对外宣传为“稳定可用的嵌入式数据库发布版”。
